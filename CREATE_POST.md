# 블로그 글 작성 가이드

이 문서는 `src/content/*.adoc` 파일들의 형식과 문체를 분석하여, 새 글 작성 시 참고할 수 있도록 정리한 것입니다.

## 1. 헤더 형식

### 필수 필드

```adoc
= 글 제목
정상혁
2026-03-28
:jbake-type: post
:jbake-status: published
:jbake-tags: tag1, tag2
:idprefix:
```

### 선택 필드

```adoc
:description: 포스트의 한줄 요약 (최근 글에는 거의 항상 포함)
:jbake-last_updated: 2026-03-28
:jbake-og: {"image": "img/topic/thumbnail.jpg"}
:toc:
:sectnums:
:toclevels: 1
:source-repo: https://github.com/benelog/example
:source-link-base: {source-repo}/tree/master
```

### 필드별 참고사항

- **태그**: 3~5개, 영문 소문자 위주 (`claude-code`, `java`, `git`, `linux`, `test`, `책`)
- **description**: 100~200자, 최근 글(2023년 이후)에는 거의 필수로 포함
- **jbake-last_updated**: 내용을 수정했을 때만 추가. 원래 작성일은 변경하지 않음
- **jbake-og**: SNS 공유 시 표시할 이미지가 있을 때 사용
- **toc**: 섹션이 3개 이상인 기술 글에는 포함 권장
- **sectnums**: 목차 번호 매기기. 단계적 설명이 많은 기술 문서에 사용

## 2. 글의 구조

### 도입부

첫 문단에서 글의 목적과 배경을 바로 밝힌다.

**기술 글 패턴:**
> Claude Code는 Anthropic의 API를 호출하는 CLI 도구입니다. Enterprise Plan에서는 API 사용량에 따라 과금되므로, 과금 구조를 이해하면 같은 작업을 더 적은 비용으로 할 수 있습니다.

**경험 공유 패턴:**
> Ubuntu 공식 홈페이지에서 ISO를 받아 같은 방법으로 설치했음에도 설치 시점에 따라서 Lenovo ThinkPad에는 OEM 커널이, Dell XPS에는 HWE 커널이 설치되는 경우를 경험했습니다.

**의견 제시 패턴:**
> Git의 브랜치 전략으로 Git-flow가 가장 유명하고 자주 언급됩니다. 그런데 이 전략은 복잡도와 실행하는 난이도가 높은 편입니다.

### 본문

- `==` (h2) 단위로 주요 섹션 구분
- `===` (h3)은 세부 분류가 필요할 때만
- 긴 글은 "배경 → 개념 설명 → 구체적 예시 → 주의사항" 순서
- 여러 방법이 있을 때는 장단점을 비교하여 제시

### 마무리

- **참고 자료** 섹션으로 마무리하는 것이 일반적
- 명시적 결론 문단은 있을 수도, 없을 수도 있음
- 댓글 유도나 공유 요청은 하지 않음

```adoc
== 참고 자료

* https://example.com[제목 또는 설명]
* https://example.com[제목 또는 설명]
```

## 3. 문체

### 어조

- **존댓말** 사용: "~입니다", "~합니다", "~할 수 있습니다"
- 개인 의견 제시 시: "저는 ~라고 생각합니다", "~라고 판단합니다"
- 경험 언급 시: "~하는 경우를 경험했습니다", "~를 몇 번 목격했습니다"

### 특징

- **실무 경험 기반**: 이론보다 직접 겪은 상황에서 출발
- **선택지 비교**: 하나의 정답을 주장하기보다 여러 방법의 장단점을 비교
- **의견은 명확히 표시**: "저는", "~라고 생각합니다"로 개인 의견임을 밝힘
- **불필요한 수식 자제**: 간결하고 담백한 서술
- 책 리뷰에서는 반말("~라고 생각한다", "공감한다")을 쓰는 경우도 있음

### 기술 용어 표기

- 영문 기술 용어는 그대로 유지: `HashMap`, `Git-flow`, `Prompt Caching`
- 한글 음차가 자연스러운 경우: "메소드", "파라미터", "애노테이션", "인터페이스"

## 4. AsciiDoc 작성 규칙

### 코드 블록

```adoc
[source,java]
----
public class Example {
    // 코드
}
----
```

- 언어를 반드시 지정: `java`, `kotlin`, `bash`, `json`, `xml`, `gradle` 등
- 에러 메시지나 실행 결과는 `[source]` (언어 미지정)로 표시
- 파일 내용을 보여주는 코드 블록에는 가급적 파일명을 소제목으로 표시:
  ```adoc
  [source,yaml]
  ..golangci.yml
  ----
  version: "2"
  ----
  ```

### 테이블

```adoc
[cols="1,2,2", options="header"]
|===
|항목 |설명1 |설명2

|값1
|내용1
|내용2
|===
```

- 비교 분석에 자주 사용

### 링크

```adoc
https://example.com[표시 텍스트]
link:relative-path.html[페이지 이동 →]
```

- 외부 링크에 `→` 화살표를 붙여 외부 이동을 암시하기도 함

### 이미지

```adoc
image::img/topic/filename.png[alt text,width=600]
```

- 이미지는 `src/content/img/<topic-slug>/` 디렉토리에 배치
- 외부 이미지 사용 시 출처를 반드시 표기: `(이미지 출처: URL)`

### 인용문

```adoc
____
인용 텍스트
____
```

- 책 인용, 다른 글의 인용에 사용

### 주의사항 블록

```adoc
[CAUTION]
====
주의 사항 내용
====

[NOTE]
====
참고 사항 내용
====
```

### 각주

```adoc
본문 텍스트 footnote:[각주 내용]
```

### 변경 이력 (장기 유지 글)

```adoc
.주요 변경이력
* 2023.03.27
** 내용 변경 설명
* 2019.07.03
** 내용 변경 설명
```

## 5. 글 유형별 패턴

### 기술 설명/가이드

- `:toc:`, `:sectnums:` 포함
- 도입부에서 문제/동기 제시
- 단계적으로 설명 (간단 → 복잡)
- 코드 블록과 실행 결과 포함
- 참고 자료 섹션으로 마무리

### 의견/분석

- 주장과 근거를 함께 제시
- 외부 사례(다른 회사, 오픈소스)를 인용
- 개인 의견임을 명확히 표시
- 비교적 짧음 (코드 블록 없이 텍스트 중심)

### 책 리뷰

- 표지 이미지로 시작
- "감상" 섹션에서 전체적 평가
- "의견 메모" 섹션에서 구절별 코멘트 (인용문 + 의견)
- "추천자료" 섹션에서 관련 자료 정리
- 인용 구절은 `____` 블록으로, 의견은 그 아래에 서술

## 6. 체크리스트

새 글 작성 시 확인할 항목:

- [ ] 헤더 필수 필드(type, status, tags, idprefix) 포함
- [ ] description 작성 (검색/SNS 공유용)
- [ ] 첫 문단에서 글의 목적이 명확한가
- [ ] 존댓말이 일관되게 사용되었는가
- [ ] 코드 블록에 언어가 지정되었는가
- [ ] 외부 이미지의 출처가 표기되었는가
- [ ] 참고 자료 섹션이 있는가
- [ ] 태그가 기존 태그와 일관성 있는가
