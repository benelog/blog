== RAG 파이프라인의 기본 흐름

전형적인 RAG는 두 시점으로 나눠 볼 수 있습니다.

**인덱싱 시점 (오프라인)**::
원본 문서 → 청킹 → 임베딩 → Vector DB에 저장. 메타데이터(출처, 작성일, 권한 등)도 함께 저장.

**질의 시점 (온라인)**::
사용자 질문 → 쿼리 재작성 → 임베딩 → Vector DB 검색 (+ 키워드 검색) → 재순위(rerank) → 컨텍스트 조립 → LLM에 프롬프트로 전달 → 응답 생성.

각 단계마다 별도의 모델이나 시스템이 끼어들 수 있고, 각각 다른 종류의 튜닝이 필요합니다.

== Vector DB는 어떤 역할을 하는가

Vector DB는 한 문장으로 말하면 **"고차원 벡터의 근사 최근접 이웃(approximate nearest neighbor, ANN) 검색에 특화된 저장소"** 입니다. 임베딩 모델이 만든 수백~수천 차원의 벡터를 저장하고, "이 쿼리 벡터와 가장 가까운 K개"를 빠르게 돌려줍니다.

핵심 책임은 세 가지입니다.

* **벡터 인덱싱**: HNSW, IVF, PQ 같은 ANN 인덱스로 검색을 sub-linear 시간에 끝냅니다. 정확도와 속도/메모리 사이에서 트레이드오프가 있습니다.
* **유사도 계산**: 코사인 유사도, 내적, L2 거리 등. 임베딩 모델이 어떤 거리를 가정하고 학습됐는지에 맞춰야 합니다.
* **메타데이터 필터링**: "2025년 이후 문서만", "이 사용자가 읽을 권한이 있는 문서만" 같은 조건을 벡터 검색과 결합. 이걸 **pre-filter / post-filter** 어느 시점에 거느냐가 성능을 크게 좌우합니다.

흔히 오해하는 지점들이 있습니다.

* **Vector DB가 의미를 이해하는 것은 아닙니다.** 의미를 벡터로 인코딩하는 일은 임베딩 모델이 합니다. Vector DB는 그 벡터끼리의 기하학적 거리만 봅니다. 검색 품질이 나쁘면 대개 임베딩 모델 쪽 문제이지 Vector DB 자체 문제는 아닙니다.
* **벡터 검색이 항상 키워드 검색보다 낫지는 않습니다.** 고유명사, 제품번호, 코드 같은 토큰은 BM25 같은 lexical 검색이 더 정확합니다. 그래서 실제 RAG는 **벡터 검색 + 키워드 검색의 하이브리드**가 기본입니다.
* **전용 Vector DB가 항상 필요하지도 않습니다.** PostgreSQL의 `pgvector`, Elasticsearch/OpenSearch의 dense vector 필드, Redis의 vector search 모듈로도 상당 규모까지 커버됩니다. 운영 부담을 줄이려면 기존 DB를 확장하는 쪽이 합리적일 때가 많습니다. Pinecone, Weaviate, Qdrant, Milvus 같은 전용 솔루션은 규모/성능 요구가 분명해질 때 고려하는 식이 무난합니다.

== 한국어 처리에서 중요한 구성요소

한국어 RAG는 영어 RAG에 비해 손이 더 많이 갑니다. 어디서 차이가 나는지 짚어봅니다.

=== 임베딩 모델의 한국어 이해 능력

가장 결정적인 부품입니다. OpenAI `text-embedding-3`, Cohere `embed-multilingual`, Voyage, BGE-M3 같은 다국어 임베딩이 점점 좋아지긴 했지만, 한국어 전용으로 튜닝된 모델(예: `KURE`, `bge-m3-korean`, 네이버 HyperCLOVA의 임베딩 등)이 도메인에 따라 더 잘 맞는 경우가 있습니다. **MTEB의 한국어 벤치마크** 점수를 참고하되, 자기 도메인 문서로 직접 평가해보는 게 가장 정확합니다.

=== 토크나이저와 형태소 분석

* **벡터 검색 쪽**: 임베딩 모델 내부의 토크나이저를 그대로 따라가게 됩니다. 사용자가 직접 손댈 여지가 적습니다.
* **키워드 검색 쪽**: 여기가 한국어에서 가장 손이 많이 가는 지점입니다. 영어는 공백 분리만으로도 어느 정도 작동하지만, 한국어는 **조사/어미가 붙은 채로 인덱싱되면 검색이 망가집니다**. "은행에서", "은행이", "은행은"이 다 다른 토큰이 되니까요.
+
Elasticsearch라면 **Nori** 분석기, OpenSearch도 동일, 그 외에 **Mecab-ko / Khaiii / KIWI** 같은 형태소 분석기를 식별기 추출/색인에 씁니다. **사용자 사전(custom dictionary)** 관리가 매우 중요합니다. 신조어, 회사 내부 용어, 제품명을 분석기가 모르면 의도와 다른 토큰으로 쪼개집니다.

=== 청킹 전략

한국어 문서는 한 문장이 길고, 접속사 없이 길게 늘어지는 경우가 많아 **고정 길이 청킹**이 의미 단위를 자주 끊습니다.

* **문장 분리(sentence splitting)** 자체가 한국어에서는 마침표 한 종류만으로 안 됩니다. "다.", "요.", "음.", "임." 같은 어미와 줄바꿈, 표/리스트 마크업까지 함께 봐야 합니다. KSS(Korean Sentence Splitter) 같은 라이브러리가 도움이 됩니다.
* **의미 단위 청킹(semantic chunking)** — 임베딩 유사도가 급격히 변하는 지점에서 자르는 방식 — 이 한국어에서 효과가 좋은 편입니다.
* **계층적 청킹**: 큰 청크로 검색하고, 매칭된 부분의 작은 청크를 LLM에 넘기는 식. 한국어처럼 문맥 의존성이 강한 언어에서 특히 유용합니다.

=== 재순위(rerank) 모델

Cross-encoder 기반의 reranker는 검색 결과를 다시 정렬해 정확도를 크게 끌어올립니다. **Cohere Rerank의 다국어 모델, BGE-reranker, Jina의 multilingual reranker** 등이 한국어를 다룹니다. 한국어 전용 reranker는 아직 선택지가 많지 않아 다국어 모델로 시작해 보고, 자체 평가셋으로 fine-tune을 검토하는 흐름이 흔합니다.

=== 쿼리 처리

사용자가 입력하는 한국어 질문은 구어체, 줄임말, 오타가 많습니다. **쿼리 재작성(query rewriting)** 또는 **HyDE(hypothetical document embedding)** 같은 기법으로 한 번 LLM을 거쳐 정제하면 검색 품질이 눈에 띄게 좋아집니다. 이때 쓰는 LLM은 비교적 작아도 충분합니다.

== RAG 안에서 쓰이는 LLM들

"RAG에 LLM 하나"라고 생각하기 쉽지만, 실제 시스템에는 **서로 다른 역할의 모델 여러 개**가 들어갑니다. 각자 요구되는 특성이 다릅니다.

[cols="1,3,2", options="header"]
|===
| 위치 | 역할 | 필요한 특성

| 임베딩 모델
| 텍스트를 벡터로 변환
| 다국어/한국어 품질, 적절한 차원 수, 빠른 추론 속도, 긴 입력 지원
| 쿼리 재작성 / 분기
| 사용자 질문을 정제하거나 어떤 도구/인덱스를 쓸지 결정
| 작고 빠른 instruction-following 모델로 충분. 지연 시간이 중요
| 재순위 모델
| 검색된 후보 K개를 정밀 점수화
| Cross-encoder. 일반 LLM이 아니라 전용으로 학습된 reranker가 효율적
| 응답 생성 LLM
| 검색된 문맥을 바탕으로 최종 답변 작성
| 긴 컨텍스트, 인용 능력, 환각 억제, 한국어 자연스러움. 대형 모델이 유리
| 평가 / 가드레일
| 응답이 컨텍스트에 충실한지, 안전한지 검증
| 비교적 작은 모델로 LLM-as-a-judge. 일관성과 비용이 중요
|===

흔한 구성은 **"쿼리 처리/판정은 작고 빠른 모델, 최종 생성은 큰 모델"** 입니다. 모든 단계를 가장 큰 모델로 돌리면 비용과 지연이 빠르게 누적됩니다. 반대로 모든 단계를 작은 모델로 돌리면 답변 품질이 무너집니다. 단계마다 적정 사이즈를 고르는 게 RAG 시스템 설계의 중요한 부분입니다.

또 한 가지, 생성 LLM이 **검색 결과를 잘 활용하는 능력은 모델마다 다릅니다**. 같은 컨텍스트를 줘도 어떤 모델은 인용에 충실하고, 어떤 모델은 자신의 사전 지식으로 답을 만들어버립니다. RAG 시스템을 평가할 때는 모델의 **groundedness**(컨텍스트 충실도)를 따로 측정하는 게 좋습니다.

== 성능이 안 나올 때 어디를 손볼 것인가

RAG 결과가 만족스럽지 않을 때, 무작정 더 큰 LLM으로 갈아끼우기 전에 **파이프라인의 어느 단계에서 망가지는지** 진단하는 게 우선입니다. 흔한 원인을 단계별로 정리합니다.

=== 1단계 — 검색이 잘못 가져온다 (retrieval failure)

가장 흔한 원인입니다. **사용자 질문에 답이 들어있는 문서를 애초에 못 찾아왔다면, 그 뒤에 어떤 LLM을 쓰든 좋아질 수 없습니다.**

진단 방법: 정답이 들어있는 청크가 top-K 안에 들어오는지 직접 확인. 들어오지 않는다면 검색 단계 문제입니다.

손볼 곳:

* **임베딩 모델 교체**: 한국어 품질이 더 좋은 모델로. 도메인 특화 모델 검토.
* **하이브리드 검색 도입**: 벡터 + BM25 결합. 고유명사/숫자 매칭이 안 되던 케이스가 풀립니다.
* **청킹 재설계**: 너무 크면 노이즈가 섞이고, 너무 작으면 문맥이 끊어집니다. 보통 한국어는 300~800 토큰 부근에서 시작해 조정.
* **메타데이터 필터**: 검색 공간을 좁히는 가장 효과적인 수단. 날짜, 출처, 권한 같은 필터를 적극 활용.
* **사용자 사전**: 키워드 검색 쪽에서 도메인 용어가 잘못 토큰화되는지 점검.

=== 2단계 — 가져오긴 했는데 순서가 나쁘다 (ranking failure)

top-20에는 정답이 있는데 top-3에는 없는 경우. LLM에 넘기는 컨텍스트가 한정돼 있으니, 순서가 결과를 좌우합니다.

손볼 곳:

* **Reranker 추가**: cross-encoder reranker가 가장 큰 효과를 내는 지점입니다.
* **쿼리 재작성**: 모호하거나 구어체인 질문을 검색에 유리한 형태로 변환.
* **다중 쿼리 (multi-query)**: 한 질문을 여러 변형으로 던져 결과를 합칩니다.

=== 3단계 — 컨텍스트는 좋은데 생성이 망가진다 (generation failure)

검색은 잘 됐고 정답 청크가 top에 있는데, 최종 응답이 틀리거나 환각이 섞이는 경우.

손볼 곳:

* **프롬프트 구조**: "주어진 문맥에만 근거해 답하라", "문맥에 없으면 모른다고 답하라" 같은 지시. 인용 형식 강제.
* **컨텍스트 배치**: 중요한 문서를 맨 앞이나 맨 뒤에. "lost in the middle" 현상을 피합니다.
* **생성 모델 업그레이드**: 여기서 더 큰 모델이 효과적입니다. 단, 1~2단계가 망가진 상태에서는 모델만 키워도 별 효과가 없습니다.
* **citation / grounded generation**: 답변마다 출처 청크를 명시하도록 강제하면 환각이 줄어듭니다.

=== 4단계 — 평가가 없어서 어디가 망가졌는지 모른다 (evaluation gap)

실은 가장 근본적인 문제입니다. **평가셋이 없으면 어떤 변경이 좋아진 것인지 알 수 없습니다.** "감으로" 튜닝하다 보면 한 쪽을 고치면서 다른 쪽이 망가집니다.

손볼 곳:

* **질문-정답-출처 셋으로 된 평가 데이터셋** 구축. 50~100개로도 시작 가능.
* **단계별 지표**: 검색 단계는 recall@K, MRR. 생성 단계는 faithfulness, answer relevance.
* **Ragas, TruLens** 같은 평가 프레임워크 활용.
* **LLM-as-a-judge**로 자동화하되, 사람이 주기적으로 보정.

== 어디부터 손댈지

새 RAG 시스템을 만들거나 기존 것을 개선할 때, 제가 보는 우선순위는 다음과 같습니다.

. **평가셋부터 만든다.** 없으면 개선이 개선인지 모릅니다.
. **하이브리드 검색을 기본으로.** 벡터만 쓰지 않습니다.
. **한국어라면 형태소 분석기와 사용자 사전 정비.** 키워드 검색이 의외로 큰 비중을 차지합니다.
. **reranker 도입.** 비용 대비 효과가 큰 편입니다.
. **생성 모델 업그레이드는 마지막에.** 가장 비싸고, 1~3이 부실하면 효과가 작습니다.

RAG의 성능 개선은 **"더 큰 모델"이 아니라 "더 정확한 검색"** 에서 오는 경우가 대부분입니다. 앞 글에서 RAG를 "프롬프트 레이어로 데이터를 흘려보내는 파이프라인"이라고 부른 이유도 여기에 있습니다. 모델은 토큰으로 들어온 것에 답할 뿐, 들어오지 않은 것에 답할 수는 없습니다.

'''
이 포스트는 Claude(claude.ai)와 정상혁이 함께 작성했습니다.
