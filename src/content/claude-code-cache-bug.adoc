2026년 3월 초부터 약 한 달간 Claude Code에 Prompt Cache가 제대로 작동하지 않는 버그가 있었습니다.
link:claude-code-cost.html[Claude Code 토큰 비용과 프롬프트 캐싱]에서 정리했듯이 Prompt Caching은 입력 토큰 비용을 90%까지 절약하는 핵심 수단인데, 이 캐시가 작동하지 않으면 비용이 크게 증가합니다.
이 글에서는 해당 기간에 발생한 두 가지 캐시 버그의 원인과 영향을 정리합니다.

== 버그 1: Resume 시 캐시 회귀

=== 원인

v2.1.69에서 도입된 `deferred_tools_delta` 기능이 원인입니다.
새 세션과 재개된 세션에서 `messages[0]`의 구조가 달라지면서, Prompt Cache의 prefix가 일치하지 않게 되었습니다.

[cols="1,2,1", options="header"]
|===
|세션 유형 |messages[0] 내용 |크기

|새 세션
|deferred tools 목록, MCP 설정, skills, 날짜 컨텍스트
|~13.4KB

|재개 세션
|날짜 컨텍스트만 포함 (나머지는 messages 끝에 추가)
|~352B
|===

Prompt Cache는 prefix의 해시값으로 캐시를 조회합니다.
`messages[0]`의 내용이 13.4KB에서 352B로 바뀌면 해시값이 완전히 달라지므로, 이전 세션에서 쌓은 캐시를 전혀 재사용할 수 없었습니다.

=== 영향 범위

이 버그는 `--resume` (CLI 플래그)과 `/resume` (슬래시 커맨드) 모두에 해당합니다.
두 방식 모두 세션 재개 시 동일한 메시지 구조 불일치가 발생하기 때문입니다.

단, Prompt Cache의 TTL이 5분이므로 실질적인 비용 손해가 발생하는 경우는 제한적입니다.

[cols="1,2", options="header"]
|===
|시나리오 |영향

|일반 연속 세션 (resume 없이)
|영향 없음. `messages[0]`이 세션 내내 안정적으로 유지

|5분 이내에 resume
|캐시가 살아있어야 하는데 prefix 불일치로 캐시 미스 발생 → **비용 손해**

|5분 이후에 resume
|어차피 TTL 만료로 캐시 소멸 → 버그와 무관하게 비용 동일

|`--print --resume` 자동화
|매 호출이 새 프로세스이므로 캐시를 전혀 재활용하지 못함 → **가장 큰 피해**
|===

=== 비용 영향

정상 상태에서는 캐시가 쌓이면서 턴이 진행될수록 메시지당 비용이 감소합니다.

----
정상 (v2.1.68):  메시지 1 → $0.15,  메시지 4 → $0.02 (캐시 적중률 증가)
버그 (v2.1.69+): 모든 메시지 → $0.35~0.40 (매번 전체 히스토리를 Cache Write)
----

`--print --resume` 워크플로우에서는 `cache_read` 토큰이 ~11,000(시스템 프롬프트 분량)에 머물고, `cache_creation` 토큰이 300,000 이상으로 치솟는 현상이 관찰되었습니다.
메시지당 약 **20배의 비용 증가**가 보고되었습니다.

== 버그 2: Tool Schema 캐시 미스

=== 원인

Claude Code가 매 턴마다 tool schema를 `JSON.stringify`로 재직렬화하여 캐시 키를 계산하고 있었습니다.
JSON 직렬화는 키 순서나 공백 등이 보장되지 않으므로 footnote:[JavaScript의 `JSON.stringify`는 객체 키 순서를 삽입 순서로 유지하지만, 내부 상태 변화에 따라 직렬화 입력 자체가 달라질 수 있습니다.], 동일한 schema라도 턴마다 다른 바이트가 생성될 수 있습니다.
바이트가 달라지면 캐시 키 해시가 달라지고, 캐시 미스가 발생합니다.

=== 영향 범위

이 버그는 resume 여부와 무관하게, **일반 연속 세션에서도 발생**할 수 있습니다.
다만 모든 사용자에게 영향을 미치는 것은 아니고, 다음 조건에 해당하는 경우에 발생합니다.

* **MCP 서버를 설정한 경우**: 외부 MCP 서버의 tool schema가 재직렬화 시 불안정
* **Deferred tools를 사용하는 경우**: `ToolSearch`로 동적 로드되는 도구의 schema가 턴마다 달라질 수 있음

세션이 길수록 schema 변동이 누적되어 비용 영향이 커집니다.
Resume 버그가 1회성 캐시 미스인 반면, 이 버그는 **매 턴마다 지속적으로** 캐시 효율을 떨어뜨립니다.

=== 비용 영향

캐시가 정상 작동하면 턴이 진행될수록 `cache_read` 토큰이 증가하고 비용이 감소해야 합니다.
이 버그가 발생하면 `cache_read` 토큰이 증가하지 않고, 매 턴 전체 히스토리가 다시 `Cache Write`로 과금됩니다.

100턴 세션 기준으로 추정하면:

----
정상:  100턴 누적 비용 ≈ $2  (캐시 적중률 95% 이상)
버그:  100턴 누적 비용 ≈ $33 (매 턴 전체 히스토리 재과금)
----

== 수정 내용 (v2.1.90)

v2.1.90 (2026-04-01)에서 두 버그가 모두 수정되었습니다.

* **Resume 버그**: 세션 재개 시 `messages[0]`에 deferred tools, MCP 설정, skills를 새 세션과 동일한 위치에 삽입하도록 변경
* **Tool Schema 버그**: tool schema를 한 번만 계산하고 캐시하여, 매 턴 재직렬화하지 않도록 변경

현재 버전 확인:

[source,bash]
----
claude --version
----

v2.1.90 이상이면 두 버그 모두 수정된 상태입니다.

== 영향 받았는지 확인하는 방법

2026년 3월 5일~4월 1일 사이에 Claude Code를 사용했다면, 다음 방법으로 영향 여부를 확인할 수 있습니다.

=== Resume 버그의 영향 여부 확인

Resume 버그는 `/resume`이나 `--resume`으로 세션을 재개했을 때, 직전 세션 종료 후 5분 이내에 재개한 경우에만 실질적인 비용 손해가 발생합니다.
`~/.claude/history.jsonl`에 `/resume` 명령 기록이 남으므로, 이를 활용하여 확인할 수 있습니다.

[source]
.Claude Code 프롬프트
----
~/.claude/history.jsonl에서 2026년 3월 5일~4월 1일 사이의 /resume 기록을 찾아줘.
각 resume 시점에 대해, 같은 프로젝트의 ~/.claude/projects/ 아래 JSONL 세션 로그에서
직전 세션의 마지막 timestamp와의 시간 간격을 계산해줘.
5분 이내에 resume한 경우가 있는지 확인하고 싶어.
----

5분 이내에 resume한 기록이 없다면, 어차피 Prompt Cache의 TTL(5분)이 만료된 후 재개한 것이므로 이 버그로 인한 추가 비용은 발생하지 않은 것입니다.

=== Tool Schema 버그의 영향 여부 확인

Tool Schema 버그는 MCP 서버나 deferred tools를 사용하는 환경에서만 발생합니다.
Claude Code 프롬프트에서 다음을 확인할 수 있습니다.

==== MCP 서버 설정 확인

MCP 서버가 설정되어 있는지 확인합니다.

[source]
.Claude Code 프롬프트
----
내 Claude Code 설정 파일들(~/.claude/settings.json, ~/.claude/settings.local.json,
프로젝트의 .claude/settings.json, .claude/settings.local.json)에
mcpServers 설정이 있는지 확인해줘.
----

`mcpServers` 설정이 없다면 MCP 서버 경로로는 이 버그의 영향을 받지 않은 것입니다.

==== Deferred tools 사용 여부 확인

Deferred tools는 MCP 서버 설정 없이도 Claude Code에 내장되어 있습니다.
해당 기간의 세션 로그에서 `ToolSearch` 호출 여부를 확인하면 됩니다.

[source]
.Claude Code 프롬프트
----
~/.claude/projects/ 디렉토리 아래의 JSONL 세션 로그 파일들 중
2026년 3월 5일~4월 1일 사이에 수정된 파일에서
ToolSearch 도구 호출이 있었는지 확인해줘.
----

`ToolSearch` 호출이 없었다면 deferred tools 경로로도 영향을 받지 않은 것입니다.
반대로, MCP 서버를 사용했거나 `ToolSearch`를 호출한 세션이 있다면 해당 세션에서 캐시 효율이 저하되었을 가능성이 있습니다.

=== API 사용량 확인

Anthropic Console에서 해당 기간의 토큰 사용량을 확인합니다.
평소 대비 `cache_creation_input_tokens`가 비정상적으로 높고 `cache_read_input_tokens`가 낮다면 이 버그의 영향을 받았을 가능성이 있습니다.

=== 캐시 메트릭 모니터링

세션 중 API 응답의 캐시 관련 토큰을 확인합니다.

* `cache_creation_input_tokens`: 대화가 진행될수록 **감소**해야 정상
* `cache_read_input_tokens`: 대화가 진행될수록 **증가**해야 정상

버그 발생 시에는 `cache_read_input_tokens`가 낮은 값에 머물고, `cache_creation_input_tokens`가 계속 높게 유지됩니다.
